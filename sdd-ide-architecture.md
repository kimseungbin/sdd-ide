# SDD IDE — 아키텍처 개요 (v1 설계)

_Spec-Driven Development IDE. Kiro 를 참조점으로 하되, 근본 설계를 뒤집은 구조._
_이 문서는 최종 설계 상태를 서술한다. 결정에 이르는 과정·근거는 별도의 decision log 참조._
_기준: brainstorm session 1 종료 시점 (결정 D1–D24)._

---

## 1. 한 문장 요약

**마크다운 파일을 진실의 원천으로 삼아 에이전트가 텍스트를 읽고 편집하는 Kiro 방식을 버리고, 타입이 있는 구조적 블록 스토어를 진실의 원천으로 삼아 사람과 에이전트가 동일한 검증된 API 를 통해서만 그것을 변경하며, Git 에 커밋되는 텍스트 투영을 durable record 로 두는 개발자 중심 SDD IDE.**

---

## 2. 핵심 테제 (왜 이게 다른가)

Kiro 의 스펙은 `requirements.md` / `design.md` / `tasks.md` — 마크다운이 곧 스펙이고, 에이전트가 그 텍스트를(체크박스 `- [ ]` 문법 포함) 직접 읽고 고친다. 이 방식의 문제는 취약성이다: 파싱/직렬화 과정에서 drift 가 생기고, 에이전트가 "텍스트를 편집"하는 것과 UI 가 보여주는 것이 어긋난다.

이 설계의 반대 베팅:

1. **구조가 진실이다** — 스펙은 타입이 있는 블록 트리다. 마크다운은 그 투영일 뿐이다.
2. **변경은 오직 API 로** — 사람이든 에이전트든 동일한 검증된 mutation API 를 통해서만 변경한다. 에이전트에게 텍스트 편집이라는 뒷문은 없다.
3. **Git 이 durable record** — 커밋되는 텍스트 투영이 durable record 이고, 라이브 스토어는 거기서 재구성(rehydrate)되는 작업 계층이다.

이 세 가지가 나머지 모든 설계를 파생시킨다.

## 3. 대상 사용자 (v1)

**개발자 중심.** v1 의 주 사용자는 개발자다. "PM 과 개발자가 스펙을 공유"하는 목표는 v1 에서 제외했다 — 실시간 협업 없이 PM 공동 저작을 강제하면 어정쩡한 비동기 참여 절충에 자원이 든다. PM 은 표준 PR 리뷰 코멘트로 관여한다. (PM 참여는 향후 확장 여지로 남김.)

## 4. 데이터 모델

### 4.1 진실의 원천과 durability

- **라이브 스토어**: 타입이 있는 블록 트리. IDE 안의 모든 것을 구동한다.
- **불변식(핵심)**: 진실은 오직 구조적 API mutation 으로만 바뀐다. 에이전트는 절대 텍스트를 편집하지 않는다.
- **Durability**: 커밋되는 텍스트 투영이 durable record 다. 라이브 스토어는 거기서 결정론적으로 재구성된다.
- 이것은 "마크다운이 진실"이 아니다 — "구조적 mutation 만이 진실을 바꾸는 유일한 방법이고, 커밋된 투영이 진실이 durable 하게 사는 곳"이다.

### 4.2 두 개의 관계 (같은 노드, 다른 그래프)

시스템은 같은 노드 위에 **두 개의 독립된 관계**를 유지한다. 이 둘을 개념적으로 분리하는 것이 UI 가 hairball 이 되는 것을 막는다.

| 관계 | 질문 | 구동하는 것 |
|---|---|---|
| **Containment (계층)** | "이 스펙은 무엇으로 이루어지는가" | 중첩, 템플릿, 드래그드롭 |
| **Dependency (그래프)** | "무엇이 무엇을 막는가" | 그래프 뷰, 지연, 포커스 |

Dependency 는 파일 경계를 넘나든다. 모든 노드가 전역 ID 를 가지므로, cross-file 의존성은 그냥 엣지 레코드 `{from, to, type}` 다 — 텍스트 관습이 아니다.

### 4.3 엣지 분류 (hard vs. soft)

네 가지 엣지 타입이 두 행동 클래스로 나뉜다. 이 분류가 시스템 전체의 척추다.

- **Hard — `blocks`, `supersedes`**: 세션이 *무엇을 할 수 있는가*를 바꾼다. 절대 컨텍스트에서 조용히 누락되면 안 된다. 누락 시 *잘못된* 행동을 유발한다.
- **Soft — `relates`, `informs`**: 세션이 *무엇을 알아야 하는가*를 바꾸지만 행동을 막지는 않는다. 양이 많으면 요약/축약 가능하다.

### 4.4 지연 결정 노드 (Deferred Decision)

일급 노드 타입이다. 작업 중 결정 사항이 떠오르면(현재 컨텍스트를 오염시킬) — 그 순간의 맥락과 의존성 엣지를 함께 포착해 활성 뷰 밖으로 치워두고, 그래프에 미해결 blocker 로 표시한다. 생명주기: open → resolved (blocks-until-resolved).

이것이 시스템의 가장 독특한 primitive다. 목적은 문서화가 아니라 **사람과 에이전트의 작업 컨텍스트를 지금 당장 풀 필요 없는 결정으로부터 보호**하는 것이다.

### 4.5 템플릿

기본 triad(requirements → design → tasks)를 제공하되(초보자용 검증된 출발점), 템플릿은 사용자 편집 가능하다. 편집 가능한 템플릿은 *구조를 정의하는 구조적 데이터*이므로 마크다운 파일이 아니라 스토어 안에 산다.

## 5. 엔진과 호출자

**하나의 spec engine, 두 종류의 호출자.**

- **spec engine**: 타입이 있는 검증된 API. 앱의 데이터 접근 계층.
- **호출자 A — 내부(in-process)**: IDE 의 UI 가 엔진을 직접 호출. 동기, 타입, 빠름, LLM 무관.
- **호출자 B — MCP 어댑터**: LLM 에이전트가 동일한 연산을 MCP 도구로 호출.

핵심 귀결: 에이전트에게 특권적 데이터 경로가 없다. 모든 mutation 이 동일한 API + 검증을 통과한다. 이것이 "구조가 진실"을 열망이 아니라 강제로 만든다.

이 단일 mutation 경로는 이후 여러 강제의 자연스러운 지점이 된다 — 역할 권한 검사, 사이클 검사가 모두 여기 얹힌다.

## 6. 컨텍스트 관리 (제품의 핵심)

### 6.1 원칙: 스토어가 막(membrane)이다

스펙 작성과 구현은 반대의 인식 모드다 (작성 = 발산적/추상적, 구현 = 수렴적/구체적). 한 컨텍스트에 섞으면 셋 다 오염된다:

- **하향 오염**: 구현 디테일이 스펙으로 새어 올라감 → 스펙이 의도가 아니라 코드를 서술.
- **상향 오염**: 스펙 심의가 코드로 새어 내려감 → 확정된 결정 재론, 과잉 설계.
- **작업 간 오염**: 한 작업의 잔여물이 무관한 작업을 편향.

**규칙**: 스펙 작성과 구현을 별도 세션으로 돌리고, 구조적 스토어만이 그 사이를 건넌다. 심의는 자기 세션에 남아 사라지고, 커밋된 구조적 상태만 막을 건넌다. 구현에서 발견된 사항은 오직 구조적 노드(예: 지연 결정 노드)로만 위로 탈출한다 — 인라인 편집이 아니라.

### 6.2 스코프드 컨텍스트 빌더 (핵심 primitive)

세션에 주입할 컨텍스트를 만들 때, 고정 hop 컷오프가 아니라 **level-of-detail 렌더러**다:

- 직접 이웃 → 상세
- 먼 노드 → 요약 (누락 아님)
- 중요한 노드 → 거리 무관 항상 표시

의존성 그래프가 곧 컨텍스트 관련성 필터다. "이 작업이 알아야 할 것"은 그래프 순회(작업 + 승인 기준 + 의존 이웃의 design 노드)이지, "스펙 파일 전부 로드"가 아니다.

**중요도의 정의** (거리를 무시하고 항상 표시할 노드):
- 파생 바닥값(구조적): hard 엣지 도달 가능성, fan-in(직접 in-degree)
- 명시 부스트: 별표/우선순위
- 지연 결정 노드는 *현재 작업 경로를 막을 때만* 중요.

이 한 빌더가 세 소비자를 섬긴다: 스펙 작성 세션, 구현 세션, 리뷰 세션. 그리고 인간용 그래프 뷰(포커스 뷰)도 같은 렌더러를 재사용한다.

### 6.3 오케스트레이션 권한은 앱 레이어가 소유

컨텍스트 격리는 모델 능력이 아니라, 컨텍스트 창을 구성하는 주체(= 우리 앱)가 보장한다. 언제 격리할지의 정책은 우리 오케스트레이션 레이어에 있지, 모델의 위임 라우팅(휴리스틱, 신뢰 불가)에 맡기지 않는다. 트리거는 우리가 정한 결정론적 이벤트다(지연 결정 발생, 백로그 스펙 변경, 작업 핸드오프).

Claude Code 서브에이전트는 Claude 사용 시 선택적 가속기이지, 아키텍처 의존성이 아니다. 프로바이더 무관(BYOK)하게 동작한다.

## 7. 에이전트 행동

### 7.1 지연 결정을 만났을 때 (정지 vs. 진행)

엣지 타입이 결정한다 (D9 재사용, 새 로직 없음):

- **Hard 엣지**로 미해결 결정에 막힘 → **정지하고 표면화**.
- **Soft 엣지** → **진행하며 표면화**.

### 7.2 주도성 (soft 케이스에서)

soft 엣지로 진행할 때, 프로젝트별 임계값이 에이전트의 주도성을 조절한다:

- 높은 임계값(보수적) → 결정을 기록만 하고 진행.
- 낮은 임계값(빠름) → 잠정 해결안을 제안하며 진행, 나중 리뷰용으로 표시.

**에스컬레이션 강제**: 오직 *명시적* 중요도(별표/우선순위)만 사람 개입을 강제한다. 파생 신호(fan-in)는 강제하지 않는다 — 노이즈 섞인 자동 추론에 강제 개입을 맡기면 예측 가능성이 무너지기 때문.

### 7.3 일관성 검사 (격리의 안전망)

막(membrane)이 구현 세션을 의도로부터 의도적으로 눈멀게 하므로, "코드가 의도에 맞는가"는 구현 세션 안에서 검사할 수 없다. 별도의 **읽기 전용 리뷰 역할**이 스펙과 코드를 모두 들고(스코프드 빌더로 슬라이스) drift 를 보고한다. 읽기 전용이므로 양쪽을 봐도 아무것도 오염시키지 않는다. 비용/지연 때문에 게이팅한다 — 값싼 작업은 건너뛰고, 고위험/high-fan-in 노드는 검사.

## 8. Git 통합

### 8.1 저장 위치

**스펙은 코드와 함께 in-repo 에 산다.** 모든 PR 동작이 Git-native 이므로, 정본 데이터를 Git 밖에 두고 동기화하면 Git 이 이미 푼 브랜칭/머징/버저닝을 재구현하게 된다. (Kiro 의 저장 위치는 옳았다 — 단 진실의 원천 모델은 아니다.)

### 8.2 커밋되는 텍스트 포맷

**하이브리드: 사람이 읽는 마크다운 + 내장된 안정적 ID 앵커.**

- ID 없는 마크다운은 탈락(추적성·라운드트립 불가).
- 정본 블록 포맷은 탈락(diff 에서 리뷰 illegible).
- 하이브리드가 legible 리뷰 + 지속적 identity + 안전한 라운드트립을 모두 만족.

라운드트립(parse-back)이 유일한 fidelity 리스크 지점이다. 리뷰어가 PR 에서 마크다운을 직접 편집하면(할 것이다) — durable record 에 대한 정당한 변경이므로 거부하지 않되, 조용히 스토어를 바꾸지도 않는다. **IDE 에서 사람 확인 후 반영(parse-back + confirmation).**

### 8.3 PR 리뷰가 하는 일 (v1 전부, 요구 강도 순)

1. **인라인 스펙 diff** — 스펙을 코드처럼 PR 에서 리뷰.
2. **추적성** — PR/커밋 ↔ 스펙 작업 노드 연결 (안정적 ID 필요).
3. **에이전트의 스펙 갱신 제안** — 리뷰 코멘트를 입력 신호로 받아 에이전트가 구조적 mutation 제안(텍스트 편집 아님).
4. **작업 완료가 머지를 게이팅** — 스펙 상태를 CI 에서 조회, "이 PR 의 blocking 작업이 다 끝났는가" 확인.

## 9. 그래프 UX

- **기본은 포커스 뷰** (현재 작업 이웃), level-of-detail 렌더러 재사용. 컨텍스트 관리 도구이지 장식이 아니다.
- **전체 프로젝트 그래프는 선택적** — 관계 발견(discovery)에 유용하나 일상 니즈가 아니므로 기본값 아님.

### 의존성 authoring

- **에이전트 제안 = 기본 경로** (낮은 마찰).
- **수동 = 항상 가능** (정밀).
- **참조 추론 = soft 엣지만** (언급 ≠ 의존).
- 에이전트가 **hard 엣지**를 만들 땐 **사람 확인 필수**.

### 사이클

- **Hard 엣지 사이클**(데드락) → 쓰기 시점에 API 가 차단.
- **Soft 엣지 사이클** → 허용(무해).
- "나중에 풀 예정"은 지연 결정 노드로 표현.

## 10. 실행 계층

### 10.1 플랫폼 셸: Electron

Node/CLI 생태계 정합성 우선. 오케스트레이션을 앱 레이어가 소유하고(CLI 서브프로세스로 Claude Code 서브에이전트 부착, Agent SDK 가 Node/TS 네이티브) — 전부 Electron 의 Node 런타임에 자연스럽다. 트레이드오프: 무거운 메모리 풋프린트를 내주고 생태계 마찰 제거를 산다.

### 10.2 인증: LLM 무관 BYOK

프로바이더 무관 BYOK(API 키). 서드파티는 Claude 구독 로그인을 쓸 수 없으므로(API 키 또는 Bedrock/Vertex 만), 앱 레이어가 프로바이더별 자격증명을 관리하고 격리를 소유한다.

### 10.3 블록 에디터: 프로토타입 후 결정

두 결선 후보를 짧은 스파이크로 검증 후 결정:
- **Slate/Plate** — schema-less 코어, 우리 비표준 모델에 마찰 최소, 협업 나중 얹기 용이. 단 beta 리스크.
- **Tiptap** — 성숙, IME 검증됨. 단 엄격한 스키마.

스파이크 검증 항목(3개, 나머지는 노이즈):
1. 한국어 IME 조합 처리 (최우선 — Slate beta 리스크의 판별점)
2. 우리 데이터 모델 얹을 때의 어댑터 마찰
3. 커스텀 블록 타입(지연 결정 노드 등) 렌더/편집

에디터는 어댑터를 통해 채택하되(직접 구축 아님), 에디터의 문서 모델은 스토어의 투영으로 취급한다. 의존성 그래프는 에디터의 일이 아니다(별도 구축) — 에디터는 containment 편집만 담당.

### 10.4 권한 강제 (프로바이더 무관)

**API 게이팅이 1차 방어선**: spec-engine API 가 역할을 인지해, 역할에 맞지 않는 mutation 을 거부(스펙 세션의 코드 쓰기, 구현 세션의 스펙 쓰기 모두 차단). 모든 쓰기가 우리 API 를 통과하므로 프로바이더 무관하게 동작한다. 샌드박스(파일시스템 격리)는 필요 시 나중에.

### 10.5 실시간 협업: v1 제외

Git 비동기 협업만. CRDT/동기 서버/프레즌스/오프라인 재동기화 전체 스택을 v1 에서 뺀다. 토대(구조적 스토어, API-only 쓰기, 전역 ID)가 CRDT-ready 이므로 나중에 재설계 없이 얹을 수 있다.

## 11. 관통하는 설계 원리

세션 전체가 하나의 패턴을 따랐다: **새 메커니즘을 발명하지 않고 이미 정한 primitive 를 재사용한다.** 두 축이 시스템을 지탱한다.

**축 1 — 엣지 분류(hard/soft)** 가 결정하는 것:
- 컨텍스트 fidelity (무엇을 절대 누락 안 할지)
- 에이전트 정지 vs. 진행
- 엣지 생성 시 신뢰 요구 수준
- 사이클 차단 여부

**축 2 — 스코프드 컨텍스트 빌더(LOD 렌더러)** 가 섬기는 것:
- 스펙 작성 / 구현 / 리뷰 세션의 컨텍스트
- 인간용 그래프 포커스 뷰

**트러스트 그래디언트**: 행동이 강할수록 신호는 엄격하게. Soft 는 에이전트 자유, hard 는 사람 게이트. 파생 신호는 "무엇을 보여줄까"에, 명시 신호만 "사람을 강제로 끼워넣을까"에.

**D2 단일 mutation 경로**: 모든 쓰기가 한 곳을 통과 → 검증, 역할 권한, 사이클 검사의 자연스러운 강제 지점.
